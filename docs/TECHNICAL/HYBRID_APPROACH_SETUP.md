# Hybrid Approach Setup Guide

## 📋 Overview

**Hybrid Approach** = Firebase Auth + Firestore + Local JSON Reference

```
┌─────────────────────────────────┐
│  Login (Email + Password)       │
│  ↓ Firebase Authentication      │
└─────────────┬───────────────────┘
              │
              ▼
┌─────────────────────────────────┐
│  Fetch Roles from Firestore     │
│  ↓ Database users/{uid}         │
└─────────────┬───────────────────┘
              │
              ▼
┌─────────────────────────────────┐
│  Set in Auth Store              │
│  ↓ Pinia useAuthStore           │
└─────────────┬───────────────────┘
              │
              ▼
┌─────────────────────────────────┐
│  Display Sidebar + Menu         │
│  ↓ Role-based UI                │
└─────────────────────────────────┘
```

## � Why Two Systems with Same Emails?

### The Design Principle: **Single Source of Truth**

**Question:** Why do we need the same email in both Firebase Auth AND Firestore?

**Answer:** They serve different purposes:

| System | Purpose | Why Needed |
|--------|---------|-----------|
| **Firebase Auth** | Secure login | Verifies password, generates JWT token |
| **Firestore** | Role/Permission data | Stores job role, permissions, employee info |

### The Problem We Solve

Without matching emails, you'd have:
```
❌ BAD: Firebase Auth has "test@example.com" but Firestore has "kook@bigt.co.th"
Result: User can log in but app can't find their role → "Unknown role" error
```

### The Solution: **Email Matching**

```
✅ GOOD: Both systems use same email "kook@bigt.co.th"
Result: After login, app finds user in Firestore and gets role → Works perfectly!
```

### Why Not Just Use Firebase Custom Claims?

We could store roles in Firebase Auth `customClaims`, but:
- ❌ Can't change roles without re-issuing tokens
- ❌ Can't store complex permission lists
- ❌ Can't use Firestore real-time sync

**So:** Firebase Auth = login only | Firestore = everything else

### Data Flow

```
Login with "kook@bigt.co.th"
        ↓
Firebase Auth: validates password, returns UID
        ↓
App queries Firestore: users/{uid}
        ↓
Finds document with email: "kook@bigt.co.th"
        ↓
Gets role: "cashier", permissions: [...], isActive: true
        ↓
Stores in Pinia useAuthStore
        ↓
Sidebar renders with correct menu
```

## 🔄 How It Works

### Login Flow
```
1. User enters email + password → /login page
2. Firebase Authentication verifies credentials
3. If valid, get Firebase UID
4. Fetch user document from Firestore: users/{uid}
5. Extract: primaryRole, roles, isActive
6. Store in Pinia useAuthStore
7. Redirect to dashboard
8. Sidebar renders based on primaryRole
```

### Role Resolution
```
// In useAuth.ts composable
const userDocRef = doc(db, 'users', user.uid)
const userDocSnap = await getDoc(userDocRef)
const { primaryRole, roles, isActive } = userDocSnap.data()
```

### Fallback
```
If Firestore document not found:
→ Fall back to extractRoleFromDisplayName()
→ (Backward compatibility)
```

## 🐛 Troubleshooting

### Problem: Login works but role = 'unknown'
**Solution:** Check Firestore document exists for that UID
```
Console → Firestore → users collection → Check user UID document
```

### Problem: Firestore seed fails
**Solution:** Check endpoint in development mode
```
1. Verify NODE_ENV !== 'production'
2. Check /api/admin/seed-firestore endpoint exists
3. Check browser console for error logs
```

### Problem: Email different in Firebase Auth vs Firestore
**Current Design:** 
- Firebase Auth stores email (verified)
- Firestore also stores email (mirror)
- Future: Can remove from Firestore if needed
